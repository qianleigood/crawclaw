# 控制 UI UX 方案

## 目标

把浏览器 Control UI 做到让第一次使用的人也能完成这 5 件事：

1. 连上 gateway
2. 连上一个渠道
3. 启用需要的 skills
4. 选对 agent
5. 发出一条成功的测试消息

当前 UI 的问题不是能力不够，而是默认界面更像工程控制台，不像面向普通用户的产品界面。

## 当前问题

从现在的源码结构来看：

- 顶层导航太多，对新手不友好。
- 设置页按系统内部结构分组，不按用户任务分组。
- Channels 和 Skills 默认暴露了太多底层状态。
- 浏览器 UI 还没有完整的一体化首次配置向导。
- 高级观测能力很强，但默认太靠前。

## 设计原则

- 默认使用 `简单模式`，显式提供 `高级模式`。
- 每个页面只回答一个核心问题。
- 优先给“下一步动作”，而不是原始状态。
- JSON、payload、诊断细节默认折叠。
- 文案尽量说人话：
  - `已连接`
  - `需要登录`
  - `可直接使用`
  - `需要配置`
- 第一屏要像工作台，不要像状态报表。

## 目标信息架构

### 简单模式

- `首页`
- `对话`
- `连接`
- `技能`
- `智能体`

### 高级模式

- `会话`
- `用量`
- `定时任务`
- `节点`
- `日志`
- `调试`
- `原始配置`

## 页面方案

## 首页

目标：告诉用户系统现在怎么样，以及接下来该做什么。

核心模块：

- `系统状态`
  - gateway 是否已连接
  - 默认 agent
  - 已连接渠道数
  - 已启用 skills 数
- `下一步`
  - 连接渠道
  - 登录 Feishu user tools
  - 启用推荐 skills
  - 发测试消息
- `快捷动作`
  - 去连接
  - 去技能
  - 去智能体
  - 发送测试消息
- `最近问题`
  - 只显示最重要的 3 条
  - 每条问题都带一个修复动作

不要做成：

- 大片原始状态墙
- 默认展示 JSON
- 每张卡片权重都一样

## 对话页

目标：保持聊天界面清楚、安静、直接可用。

默认可见：

- 消息历史
- 输入框
- 附件
- 当前 agent 选择
- `新会话`
- `查看本次运行详情`

默认折叠：

- Action Feed
- tool stream 明细
- 内部调试信息

规则：

- 先让它像聊天工作区
- 再让它像调试台

## 连接页

目标：让配置过程不依赖 CLI 知识。

这个页面应该把现在的登录页和渠道配置心智合并成一个地方。

建议分 3 块：

- `Gateway 连接`
  - 地址
  - token 或 password
  - 连接按钮
- `渠道连接`
  - Feishu
  - Telegram
  - Discord
  - Slack
  - 其他按需补充
- `身份连接`
  - `Feishu Bot`
  - `Feishu User`
  - 明确说明这两个不是同一个身份

每张渠道卡片只展示：

- 当前状态
- 已完成什么
- 还缺什么
- 一个主要下一步按钮

高级细节：

- raw payload
- 完整 snapshot
- 底层故障细节

全部放到 `显示高级信息` 折叠区。

## 技能页

目标：把 skill 管理变成“能力中心”。

建议分组：

- `推荐启用`
- `已启用`
- `需要配置`
- `可选扩展`

每张 skill 卡片只显示：

- 名称
- 一句话用途
- 前置要求
- 当前状态
- 一个主要动作：
  - `启用`
  - `配置`
  - `安装`
  - `查看详情`

不要默认把这些放到前台：

- `eligible`
- `bundled`
- `blockedByAllowlist`
- 原始 source 标识

这些可以放在详情抽屉里。

## 智能体页

目标：先帮助用户理解和选择智能体，而不是默认展示内部运行细节。

默认展示：

- 默认智能体
- 每个智能体适合做什么
- 使用哪些技能
- 适合哪些渠道
- 切换默认智能体

高级面板：

- `Inspect`
- timeline
- query context
- provider request
- channel streaming 状态

规则：

- 小白先看到“怎么用”
- 高级观测继续保留，但不要当作默认主入口

## 设置页

目标：按用户任务组织配置。

建议分成：

- `基础`
- `渠道与账号`
- `模型与智能体`
- `技能与工具`
- `记忆与隐私`
- `高级`

以下内容全部收进 `高级`：

- raw config
- diagnostics
- debug
- logs
- bindings
- wizard state
- MCP 和服务端内部细节

## 首次使用流程

浏览器 UI 需要有完整的首次向导，而不只是一个特殊模式。

推荐流程：

1. 连接 gateway
2. 选择默认 agent
3. 连接一个渠道
4. 启用推荐 skills
5. 发送测试消息

CLI 里的 setup/onboard 流程仍然可以作为逻辑参考，但浏览器端要把这套体验真正做成可视化向导。

## 视觉方向

视觉主张：`安静工作台`

建议：

- 先用留白和排版，而不是先加边框和卡片
- 优先用分区布局，不做密集卡片墙
- 全局只保留一个强调色
- 辅助文案尽量短
- 警告和错误颜色只用于真正的运行状态

动效：

- 轻量页面切换
- 清楚的保存反馈
- 连接成功/失败状态平滑切换
- 不做纯装饰动效

## 动效系统

动效的作用应该是提升层级感、确认状态变化、帮助用户建立方向感。

界面可以更高级，但不能更吵。

### 动效规则

- 动效时间要短
- 不要多种动画叠加同时出现
- 尊重 `prefers-reduced-motion`
- 动效用于确认状态变化，不用于装饰空白区域
- 重要信息不能依赖延迟入场才能看到

### 推荐动效模式

#### 首页

- 主状态与下一步动作做分层进入
- 某个阻塞项恢复正常时，做轻微高亮脉冲
- 小范围数字变化可以做轻量计数，不要全页面一起跳

#### 连接页

- 渠道卡片的状态切换：
  - `未连接 -> 连接中 -> 已就绪`
- 成功状态更适合做平滑变形，而不是大量 toast
- 连接步骤可以用进度线或当前步骤标记

#### 技能页

- 过滤切换时做柔和的重排，不要硬切
- 技能详情可以做侧滑或淡入弹层
- 启用/安装动作要有内联进度和完成反馈

#### 智能体页

- 切换 agent 时，用轻量内容过渡保持方向感
- Inspect timeline 在轮询更新时逐步显现，不要整块突然跳出

#### 对话页

- 流式输出要稳定、可读
- tool/action 区域展开和收起可用高度 + 透明度过渡
- 消息主列不要做强动画

### 适合做高级动效的地方

- 主页面切换
- onboarding 步骤切换
- 渠道连接卡片状态变化
- inspect timeline 渐进出现
- 保存 / 应用配置后的确认反馈

### 不适合做高级动效的地方

- 页面上每一张卡片
- 每一条消息气泡
- 大面积装饰性背景运动
- 操作页面上的长篇 hero 动画

### 实现时可用的 skills

- `frontend-skill`
  - 用来约束视觉风格、页面层级、动效节奏
- `playwright`
  - 用真实浏览器去验证动效是否稳定、是否影响可用性

## 实施顺序

1. 精简导航，加入 `简单模式` / `高级模式`
2. 重做 `首页`
3. 合并登录与渠道设置，形成 `连接页`
4. 重做 `技能页`
5. 重组 `设置页`
6. 补完整浏览器首次向导

## 完成标准

当一个非技术用户可以只通过浏览器完成配置，而不需要阅读 raw config 或 CLI 文档时，这次重构才算完成。

也就是他能：

- 连上系统
- 配好一个可用渠道
- 启用正确的 skills
- 选对 agent
- 用一条测试消息验证成功
